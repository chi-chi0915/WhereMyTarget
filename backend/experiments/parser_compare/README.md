## Parser 비교

논문 PDF RAG에서 검색 품질은 단순 embedding 성능뿐 아니라, 문서를 RAG 입력에 적합한 구조로 변환하는 parsing 품질에 크게 영향을 받는다고 판단했다.

특히 논문 PDF는 2단 컬럼, 표, 그림, 캡션, 섹션 제목 등의 구조가 존재하기 때문에 텍스트 추출만으로는 읽는 순서가 섞이거나 표의 형태를 잃어버릴 수 있었다. 따라서 여러 parser를 동일 PDF에 적용해보고, RAG 입력으로 사용 가능한지를 비교했다.


## Parser별 비교 결과

### PyMuPDF-baseline

- 파싱 속도가 빠르다.
- 2단으로 되어 있는 문서의 경우 글의 순서가 심하게 섞였다.
- 표의 구조를 유지할 수 없었다.

결론: 1단으로 구성된 텍스트 문서에는 유효하나 범용성이 부족함


### PyMuPDF4LLM

- 파싱 속도가 느리다. 
- 2단으로 되어 있는 문서의 경우 글의 순서가 조금 섞였다.
- 표의 구조를 유지하며 Markdown 형태로 결과를 얻을 수 있었다.

결론: 파싱 품질은 믿을만 하나 글의 순서가 섞이는 부분에서 아쉬움


### PDFPlumber

- 파싱 속도가 빠르다.
- 2단으로 되어 있는 문서의 경우 글의 순서가 많이 섞였다.
- 표의 구조를 유지할 수 있었지만 Markdown 형태가 아니었다.

결론: 속도는 괜찮으나 파싱 품질 면에서 아쉬움


### Unstructured fast

- 파싱 속도가 조금 느리다.
- 2단으로 되어 있는 문서의 경우 글의 순서가 많이 섞였다.
- 표의 구조를 유지할 수 없었다.

결론: 속도에 비해 글의 순서가 섞이고 표의 구조를 유지 못하는 부분이 아쉬움


### Docling

- 파싱 속도가 빠르다.
- 2단으로 되어 있는 문서의 경우 글의 순서가 가장 섞이지 않았다.
- 표의 구조를 유지하며 Markdown 형태로 결과를 얻을 수 있었다.
- 일부 이미지의 제목 등 내용이 빠진 부분이 있어 보임

결론: 내용이 살짝 빠진 부분은 아쉽지만 글의 순서가 섞이지 않은 점을 높게 삼

### 선택

현재 실험 결과 기준으로는 Docling을 메인 parser로 선택하는 것이 가장 적절하다고 판단했다. Docling은 논문의 구조 유지, 읽는 순서 보존, 표 구조 보존 부분에서 가장 안정적인 결과를 보였다. 다만 Docling도 이미지 내부 의미를 완전히 이해하는 것은 아니므로, MVP에서는 figure caption과 주변 문단을 기반으로 이미지 관련 정보를 검색 대상으로 포함한다.
